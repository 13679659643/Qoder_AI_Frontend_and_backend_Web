# 数据建模规范

## 1. 模型架构

### 星型模型优先
- 所有模型必须基于星型模型（Star Schema）设计
- 事实表（Fact）存放可度量的业务事件
- 维度表（Dim）存放描述性属性
- 仅在确有必要时使用雪花型（需说明原因）

### 表类型标识
```
Fact_xxx  — 事实表（如 Fact_Sales, Fact_Orders）
Dim_xxx   — 维度表（如 Dim_Date, Dim_Product）
Bridge_xxx — 桥接表（多对多关系，如 Bridge_CustomerProduct）
CT_xxx    — 计算表（如 CT_DateRange）
_xxx      — 辅助/隐藏表（如 _Measures, _Parameters）
```

## 2. 关系设计

### 基本原则
- 所有关系必须是 1:N（一对多），从维度表指向事实表
- 筛选方向默认单向（维度 → 事实）
- 双向筛选必须有明确业务理由，并在文档中说明
- 禁止循环依赖
- 每个事实表必须关联日期维度表

### 日期表要求
- 必须有独立的日期维度表（不使用自动日期表）
- 日期表必须包含完整连续的日期范围
- 必须标记为日期表（Mark as Date Table）
- 包含标准层级：Year → Quarter → Month → Week → Day

### 关系文档化
每条关系必须记录：
```
源表.源列 → 目标表.目标列 | 基数 | 筛选方向 | 是否活跃 | 业务含义
```

## 3. 表设计规范

### 事实表
- 只保留外键和度量值列
- 描述性属性放维度表，不冗余到事实表
- 大型事实表考虑增量刷新

### 维度表
- 包含代理键（Surrogate Key）和业务键
- 包含所有描述性属性
- 小型维度表使用 Dual 存储模式（DirectQuery 场景）

### 列优化
- 移除所有未使用的列（减小模型体积）
- 文本列检查是否可以用整数编码替代
- 避免高基数文本列（超过 100 万唯一值需评估）
- 日期列统一为 Date 类型（不使用 DateTime 除非必要）
- 数值列选择最小精度（整数优于小数）

## 4. 度量值组织

### Display Folder 分组
```
Measures/
├── Base Metrics/          — 基础度量值（SUM, COUNT, AVERAGE）
├── Time Intelligence/     — 时间智能度量值（YTD, QTD, MoM）
├── Ratios & KPIs/         — 比率和关键指标
├── Rankings/              — 排名度量值
├── Formatting/            — 格式化辅助度量值
└── _Internal/             — 内部辅助度量值（隐藏）
```

### 度量值表
- 创建专用空表 `_Measures` 存放所有度量值
- 或按业务域分多个度量值表（大型项目）

## 5. 禁止事项

- ❌ 禁止使用自动日期/时间表（在选项中关闭）
- ❌ 禁止在事实表之间建立直接关系
- ❌ 禁止使用多对多关系除非通过桥接表
- ❌ 禁止在模型中保留未使用的表或列
- ❌ 禁止使用 Power BI 自动生成的隐藏日期层级
